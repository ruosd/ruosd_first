# 第三阶段架构升级：执行计划

## 步骤选择原则

第三阶段共 5 个改进方向，按 **价值/改动量比** 和 **依赖关系** 排序后，先做前 3 步：

| 顺序 | 步骤 | 原因 |
|------|------|------|
| 1 | 多租户用户隔离 | 基础中的基础：用户数据不隔离，后续所有功能都是共享的 |
| 2 | 对话自动归档 | 改动小、见效快：超长对话摘要压缩，省 token 省存储 |
| 3 | 异步文件上传 | 解决实际痛点：大文件上传不阻塞 HTTP 请求 |

4~5 步（知识库版本、LangGraph）留到后续迭代。

## 为什么"多租户隔离"排第一

类比：你现在住的楼，所有住户共用一把大门钥匙、共用一个邮箱。虽然能用，但 A 的快递会被 B 拿走，C 能看 D 的账单。

当前项目的知识库、对话记忆都没有按用户隔离。不同用户的聊天记录虽然 session_id 不同，但向量检索时搜的是同一个全局知识库。这意味着：
- 用户 A 上传的内部文档，用户 B 也能搜到
- 管理员看记忆统计时，混在一起无法区分

先做隔离，后续所有功能才能建立在正确的地基上。
