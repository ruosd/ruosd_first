# 多Agent电商客服系统 — 企业级改进策略全录

> 共三阶段 15 步骤，改动约 25 个文件，覆盖安全、防御、可靠性、可观测、工程化、架构六个维度。

---

## 第一阶段：安全加固 + 可靠性

### 1. JWT 密钥环境变量化

| 项 | 内容 |
|---|------|
| **问题** | 密钥硬编码 `"your-secret-key-change-in-production-2026"`，任何人看到源码即可伪造任意用户令牌 |
| **策略** | 密钥从代码剥离，存入 `.env`，`os.getenv()` 读取。12-Factor App 原则：代码与配置分离 |
| **原理** | JWT 安全性完全依赖密钥保密性，密钥泄露 = 认证体系瓦解。环境变量是容器化部署的标准配置方式 |
| **改动** | `.env`、`admin_router.py`、`user_router.py`、`docker-compose.yml` |

### 2. 密码哈希 MD5 → bcrypt

| 项 | 内容 |
|---|------|
| **问题** | 管理员用 MD5 无盐，用户用 SHA256+固定盐。MD5 每秒可算百亿次，彩虹表秒查 |
| **策略** | 创建 `src/utils/password.py` 统一入口，全部替换为 bcrypt（12轮迭代） |
| **原理** | bcrypt 三合一：自动随机盐（相同密码不同哈希）+ 故意慢（0.3秒/次，暴力成本极高）+ 抗 GPU（算法限制并行）。选择 bcrypt 而非 scrypt/argon2 是因为 Python 生态最成熟 |
| **改动** | 新建 `password.py`、改 `admin_router.py`、`user_service.py`、清 `hashlib` 导入 |

### 3. API 请求频率限制

| 项 | 内容 |
|---|------|
| **问题** | 无频率控制，攻击者可无限调用登录接口暴力破解 |
| **策略** | `slowapi` 基于客户端 IP 令牌桶算法，登录 5次/分、上传 10次/分、聊天 30次/分 |
| **原理** | 每个 IP 有一个"桶"，每分钟自动补充令牌。请求消耗一个，桶空返回 429。选择 slowapi 而非自己实现是因为它与 FastAPI 原生集成，支持装饰器语法 |
| **改动** | `main.py` + 三个 router |

### 4. CORS 来源白名单

| 项 | 内容 |
|---|------|
| **问题** | `allow_origins=["*"]` 允许任意网站调用 API |
| **策略** | 从环境变量读取域名白名单，限制 HTTP 方法为 GET/POST/PUT/DELETE |
| **原理** | CORS 是浏览器安全机制。`*` 等于关闭保护。限制 origin + method 双重防御 |
| **改动** | `.env`、`main.py`、`docker-compose.yml` |

### 5. 输入校验强化

| 项 | 内容 |
|---|------|
| **问题** | Pydantic 只有类型校验，无长度约束。10MB 字符串可直达 LLM |
| **策略** | 所有 `Field()` 添加 `min_length`/`max_length`，在数据进入业务逻辑前拦截 |
| **原理** | 纵深防御第一层。FastAPI 收到请求 → Pydantic 校验 → 不通过即 422，不进入后续代码。防止内存耗尽、日志洪泛、数据库溢出 |
| **改动** | 5 个文件的 Pydantic 模型全面加固 |

### 6. Ollama 失效即拒绝

| 项 | 内容 |
|---|------|
| **问题** | Ollama 不可用时静默写入零向量，上传返回成功但数据损坏 |
| **策略** | 上传前预检 → 不可用直接 503；`chroma_store_service` 二层防线全部失败时拒绝写入 |
| **原理** | Fail-fast 原则：外部依赖故障立即报告，不让系统进入不一致状态。选择 503 而非 500 是因为语义准确——告诉调用方"不是你的错，稍后重试" |
| **改动** | `admin_router.py`、`chroma_store_service.py` |

### 7. LLM 调用自动重试

| 项 | 内容 |
|---|------|
| **问题** | 网络抖动导致单次 LLM 调用失败即报错 |
| **策略** | `tenacity` 指数退避重试：3 次，1s→2s→4s，仅对网络异常重试 |
| **原理** | 只重试 `APITimeoutError`/`APIConnectionError` 等临时故障，API Key 错误等永久性故障不重试。`reraise=True` 确保 3 次全失败后抛出原始异常 |
| **改动** | `llm_service.py` 两个方法各加 `@retry` 装饰器 |

## 第二阶段：可观测性 + 工程化

### 8. 结构化日志

| 项 | 内容 |
|---|------|
| **问题** | 纯文本日志难以用工具检索分析 |
| **策略** | 双格式输出：控制台人类可读文本 + 文件 JSON 格式（每行一条） |
| **原理** | JSON 日志可用 `jq`/`grep`/Loki 高效查询。不需要引入新依赖（Python 标准库 `logging.Formatter` 即可）。WARNING 及以上自动附带 `location` 字段 |
| **改动** | `logger.py` 新建 `JSONFormatter` |

### 9. Prometheus 指标

| 项 | 内容 |
|---|------|
| **问题** | 无指标暴露，无法监控请求量、延迟、错误率 |
| **策略** | `prometheus-fastapi-instrumentator` 自动暴露 `/metrics`，追踪请求数/延迟/状态码 |
| **原理** | 一行代码 `Instrumentator().instrument(app).expose(app)` 即可。选择这个库是因为它自动追踪所有 FastAPI 端点，无需手动埋点 |
| **改动** | `requirements.txt`、`main.py` |

### 10. 健康检查强化

| 项 | 内容 |
|---|------|
| **问题** | `/health` 只返回静态 `"healthy"`，不检查依赖 |
| **策略** | 逐项探测：SQLite、Redis、ChromaDB、Ollama、MySQL，返回各自状态 |
| **原理** | 整体状态判定：全部 up → healthy；1-2 个 down → degraded；3+ → unhealthy。不返回 503 以避免负载均衡误踢 |
| **改动** | `main.py` `/health` 端点重写 |

### 11. 统一错误码

| 项 | 内容 |
|---|------|
| **问题** | 前端靠解析 `detail` 文本判断错误类型，改一个字就失效 |
| **策略** | 定义 15 个 `ErrorCode` 枚举 + `AppError` 异常类，前端按 `code` 处理 |
| **原理** | `code` 是机器可读的约定，不随文案变化而变化。全局异常处理器自动将 `AppError` 转为 `{"code": "...", "message": "..."}` 格式 |
| **改动** | 新建 `errors.py`、`main.py` 异常处理器、`admin_router.py` 改用 `AppError` |

### 12. 代码质量配置

| 项 | 内容 |
|---|------|
| **问题** | 无自动化代码检查，风格不统一 |
| **策略** | `ruff` 替代 flake8+isort+black，一个工具完成检查+格式化 |
| **原理** | 选择 ruff 而非 flake8 是因为速度（Rust 实现，毫秒级扫描）。规则集选择：F（未使用导入）、B（常见 bug）、I（导入排序）、SIM（代码简化）、UP（新语法建议） |
| **改动** | `requirements.txt`、新建 `pyproject.toml` |

## 第三阶段：架构升级

### 13. 多租户用户隔离

| 项 | 内容 |
|---|------|
| **问题** | 所有用户的对话记忆在同一个 ChromaDB 集合中，A 能搜到 B 的记忆 |
| **策略** | 通过 ChromaDB 元数据 `user_id` 字段实现逻辑隔离：写入时标记，检索时过滤 |
| **原理** | 不改集合结构（代价太大）。利用 `save_memory` 已存的 `user_id` 字段 + `retrieve_memory` 已有的 `filter_dict` 机制。唯一缺失的是入口处未从 JWT 令牌提取用户身份 |
| **改动** | `chat_router.py` 新增 `_extract_user_id()`，创建会话/发送消息自动注入 |

### 14. 对话自动归档

| 项 | 内容 |
|---|------|
| **问题** | 超长对话消耗大量 LLM token，成本高、速度慢、可能超出上下文窗口 |
| **策略** | 超过 3000 token 时用 LLM 将旧消息摘要为 1-2 句话，保留最近 6 条原文 |
| **原理** | 选择 LLM 摘要而非简单截断，因为截断会丢失关键信息（用户诉求、已解决问题）。无 LLM 时降级为截断 |
| **改动** | 新建 `history_compressor.py`、`chat_router.py` 在发送前调用 |

### 15. 异步文件上传

| 项 | 内容 |
|---|------|
| **问题** | 大文件上传同步等待向量生成，HTTP 请求阻塞 30s+ 可能超时 |
| **策略** | Celery + Redis 任务队列：API 立即返回 task_id，Worker 后台处理，前端轮询进度 |
| **原理** | Redis 作为消息代理（已有，复用）。选择 Celery 而非 FastAPI BackgroundTasks 因为：支持重试、任务持久化、进度追踪、Worker 独立扩缩容 |
| **改动** | 新建 `src/tasks/`、`admin_router.py` 新增异步端点、`docker-compose.yml` 新增 worker 服务 |

## 策略选择原则

每步决策遵循以下优先级：

1. **最小改动量** — 能用 3 行解决的不写 30 行；能用现有依赖的不新增
2. **向后兼容** — 每个改动独立可测试，不破坏现有功能，出问题可快速回滚
3. **渐进式** — 安全第一（防外部攻击）→ 防御层（限流校验）→ 可靠性（重试熔断）→ 可观测（日志指标）→ 架构（隔离归档异步）
4. **可观测** — 关键改动都有日志输出，方便定位问题
5. **可配置** — 阈值、开关都通过环境变量控制，不硬编码
